# 深圳科飞真实供应商接入设计

## 1. 背景

当前项目的供应商能力已经具备统一的主数据、配置、商品同步、提单、查单、回调和 Worker 调度骨架，但实际协议适配层仍只有 `mock-supplier`。为了让系统能够在真实环境中完成“账户余额查询、同步产品列表、话费充值”三类能力，并让新供应商成为现有充值主链路中的主要供应商，需要在不破坏现有模块边界的前提下补齐真实协议适配与手工联调入口。

本次接入的真实供应商业务名称为“深圳科飞”，上游给出的接入账户信息如下：

- 账号：`JG18948358181`
- 账号名：`金骏通信话费充值`
- 密钥：`F29C80BB80EA32D4`
- API 地址：`http://api.sohan.hk:50080/API`

对接后的部署与真实联调将发生在已完成白名单加白的服务器环境中，不以当前本机作为最终出网环境。公网回调地址预计挂载在 `admin.miigo.cn` 下。

## 2. 目标

本次设计的目标是让“深圳科飞”作为正式供应商接入现有系统，并满足以下交付要求：

1. 通过后台或内部接口可真实查询深圳科飞账户余额。
2. 通过后台或内部接口可真实同步深圳科飞的话费商品目录并回写平台商品与映射。
3. 通过现有开放下单主链路可真实发起话费充值，并由深圳科飞完成提单、回调或查单推进。
4. 接入后可通过商品映射和优先级控制，使深圳科飞成为目标商品的主要供应商，而不是只作为旁路测试供应商。

## 3. 非目标

本次设计明确不包含以下内容：

- 不接入流量充值、权益充值、游戏充值、预下单等非本期能力。
- 不重构订单、账务、通知、Worker 等主链路架构。
- 不把所有商品一次性全量切到深圳科飞。
- 不在本轮解决本机不具备白名单出网能力的问题。

## 4. 总体方案

本次采用“分层接入 + 受控切主”方案。

### 4.1 总体原则

- 协议差异全部收敛在供应商适配层。
- 订单服务、账务服务、通知服务、Worker 服务只消费平台统一事件，不感知上游协议细节。
- 真实接入后由商品映射与优先级控制深圳科飞是否成为主供应商，不通过硬编码全量切换。
- 真实回调与查单补偿同时保留，回调优先，轮询兜底。

### 4.2 模块边界

改动范围集中在以下四层：

1. 供应商协议适配层  
   在 `backend/src/modules/suppliers/adapters/` 下新增深圳科飞适配器，负责 GBK 编解码、MD5 签名、余额查询、商品列表查询、话费充值、订单查询、form-urlencoded 回调解析与状态映射。

2. 供应商服务层  
   由 `SuppliersService` 统一负责选择适配器、触发商品同步、执行提单与查单、解析回调，并向订单服务发布标准领域事件。

3. 后台/内部联调入口  
   增加后台或内部“查余额”“手工触发商品同步”的能力；真实充值仍复用现有开放下单接口，不新增专门的真实充值测试接口。

4. 商品映射与主路由控制  
   继续利用 `product.product_supplier_mappings` 中的 `priority / status / sales_status / supplier_product_code / cost_price` 控制深圳科飞是否作为目标商品的主供应商。

## 5. 协议设计

### 5.1 请求协议

深圳科飞接口统一采用以下协议：

- 请求方式：`POST`
- URL：`http://api.sohan.hk:50080/API`
- Content-Type：`application/json;charset=GBK`
- 字符集：请求与响应均按 GBK 处理
- 公共字段：
  - `sign`
  - `agentAccount`
  - `busiBody`

签名规则为：

- `sign = md5(busiBody字符串 + md5Key)`
- `busiBody` 参与签名的字符串必须与实际发送内容完全一致
- 若字段如 `retUrl`、`ispName` 需要 URL 编码，则请求体和签名串必须保持一致

这要求新增一套供应商专用的：

- 固定字段顺序序列化逻辑
- GBK 编码转换逻辑
- MD5 签名逻辑
- 供应商响应解码与字段提取逻辑

### 5.2 本期接入的业务接口

本期只接以下四个接口：

- 余额查询：`action = YE`
- 产品列表查询：`action = CHECK_SPU`
- 话费充值：`action = CZ`
- 订单查询：`action = CX`

明确不接：

- 流量充值
- 权益充值
- 游戏充值
- 预下单

## 6. 数据模型设计

### 6.1 供应商主数据

继续复用现有表：

- `supplier.suppliers`
- `supplier.supplier_configs`

新增供应商主数据时，建议固化以下字段：

- `supplierCode = shenzhen-kefei`
- `supplierName = 深圳科飞`
- `protocolType = SOHAN_API`

### 6.2 供应商配置

不新增新表，继续复用 `supplier_configs`：

- `credential`：存加密后的上游账号和密钥原始串或结构化凭证
- `callbackSecret`：存回调验签所需密钥
- `timeoutMs`：存真实调用超时
- `configJson`：存协议细节，例如：
  - `baseUrl`
  - `agentAccount`
  - `md5Key`
  - `callbackUrl`
  - `encoding = GBK`
  - `catalogSyncOptions`

### 6.3 商品同步落库

继续复用：

- `product.recharge_products`
- `product.product_supplier_mappings`

建议映射规则如下：

- `supplierProductCode`：直接保存上游 `productSn`
- `carrierCode`：将 `移动 / 联通 / 电信 / 广电` 规范映射为平台运营商枚举
- `provinceName`：保存上游省份，必要时保留 `全国`
- `faceValue`：使用上游 `amount`
- `purchasePrice`：依据折扣或实际采购价规则计算
- `rechargeMode`：本期先映射为平台当前支持的话费商品口径，不扩展到其他品类

本期重点是“同步并形成可控映射”，不是“自动把所有新商品全部上架”。

## 7. 平台接口设计

### 7.1 后台接口

为支持手工联调，新增以下后台接口：

- `GET /admin/suppliers/:supplierId/balance`
  - 用于查询深圳科飞真实账户余额
  - 返回上游账户名、预存余额、佣金余额、错误码、错误描述

- `POST /admin/suppliers/:supplierId/catalog/sync`
  - 用于手工触发话费商品目录同步
  - 由系统发起真实 `CHECK_SPU` 请求，回写商品与映射，并记录同步日志

- `GET /admin/suppliers/:supplierId/sync-logs`
  - 用于查看最近商品同步结果

### 7.2 内部接口

可以保留内部同步入口供脚本或 Worker 调用，但本期验收核心不依赖新增内部测试接口。余额查询和产品同步主要通过后台接口完成即可。

### 7.3 开放下单接口

真实话费充值继续复用现有：

- `POST /open-api/orders`

订单命中商品后，由平台候选映射与优先级选择深圳科飞作为主供应商执行提单。

## 8. 供应商适配器设计

### 8.1 新增适配器

新增一个深圳科飞适配器，建议放在：

- `backend/src/modules/suppliers/adapters/shenzhen-kefei.adapter.ts`

适配器至少需要支持以下能力：

- `getBalance()`
- `syncCatalog()`
- `submitOrder()`
- `queryOrder()`
- `parseCallback()`

由于现有 `SupplierAdapter` 接口只覆盖提单、查单、回调，设计上需要扩展适配器接口，以支持余额查询与目录同步的真实能力。

### 8.2 请求封装

适配器内部负责：

- 构造 `busiBody`
- 生成与发送完全一致的签名串
- GBK 编码请求体
- 解析并转换上游响应
- 输出平台可消费的标准结果

### 8.3 错误码映射

按当前文档，关键错误码建议映射如下：

- `1`：调用成功
- `-100`：账号不存在，视为配置错误
- `-101`：签名错误，视为协议或密钥错误
- `-102`：账户暂停使用，视为供应商配置异常
- `-103`：无 API 接口权限，视为供应商权限异常
- `-301`：IP 鉴权失败，视为环境配置异常
- `-302`：账户没有交易权限，视为供应商权限异常
- `-305`：账户余额不足，视为业务失败
- `-310`：订单号重复，不直接失败，转为查单确认
- `-201`：订单号不存在，结合时间窗口判断失败

## 9. 状态机与事件映射

### 9.1 供应商状态映射

上游订单状态码映射为平台标准状态：

- `0 / 1 / 2 / 6` -> `QUERYING`
- `11 / 16` -> `SUCCESS`
- `20 / 21 / 26 / 35` -> `FAIL`

### 9.2 提单结果处理

对于话费充值接口：

- 若调用返回 `errorCode = 1` 且生成 `chargeId`
  - 记录供应商订单号
  - 平台侧先推进到 `ACCEPTED`
  - 后续依赖回调或查单得到终态

- 若返回 `-310`
  - 不直接失败
  - 进入查单路径确认该订单是否已受理或已终态

- 若返回明确失败码且无受理依据
  - 直接映射为平台 `FAIL`

### 9.3 回调与查单协同

本次接入同时保留：

- 回调优先
- 查单补偿

即使回调暂时未到，也可通过现有 Worker 查单节奏推进订单。

## 10. 回调设计

### 10.1 协议特征

深圳科飞回调为：

- `application/x-www-form-urlencoded`
- 回调内容按 GBK 解码
- 成功响应必须返回纯文本大写 `OK`

### 10.2 验签规则

回调签名基于：

- `Orderid`
- `Chargeid`
- `Orderstatu_int`
- `Errorcode`
- `Password = md5Key`

签名串类似：

`Orderid=...&Chargeid=...&Orderstatu_int=...&Errorcode=0000&Password=<md5Key>`

然后对整个字符串做 `MD5`。

### 10.3 平台适配策略

当前统一回调入口仍保持：

- `/callbacks/suppliers/:supplierCode`

但回调解析与验签必须按供应商协议类型分支：

- `mock-supplier`：继续走现有 JSON + HMAC
- `shenzhen-kefei`：走 form-urlencoded + MD5

对深圳科飞回调路径，平台最终响应体必须返回：

- `OK`

而不是当前统一 JSON 格式。

## 11. 主路由切换策略

深圳科飞接入后，不做硬编码“全量替换所有供应商”，而是通过 `product_supplier_mappings` 逐步切主：

1. 先同步商品目录并生成映射
2. 对目标商品确认 `supplierProductCode / price / province / carrier`
3. 将深圳科飞映射配置为有效状态
4. 通过 `priority` 让深圳科飞成为目标商品主映射
5. 部署后通过手工真实充值验证成功，再逐步扩大映射范围

这样既满足“作为主要供应商参与现有充值主链路”，又保留回退能力。

## 12. 手工验收链路

部署到白名单服务器并完成公网回调配置后，手工验收按以下顺序执行：

1. 供应商余额查询
   - 调用后台余额接口
   - 确认账户名、预存余额、错误码正常

2. 商品同步
   - 调用后台商品同步接口
   - 检查 `recharge_products` 与 `product_supplier_mappings` 是否更新
   - 确认目标商品已将深圳科飞配置为主映射

3. 真实话费充值
   - 调用现有开放下单接口
   - 订单命中深圳科飞映射并完成真实提单
   - 回调或查单推进订单到终态
   - 核对订单状态、账务扣款或退款、通知状态

## 13. 风险控制

本次接入需要明确以下风险控制：

- 只接话费充值，不接其他品类
- 切主按商品维度进行，不做一键全量切换
- 回调与查单同时保留，避免单一推进路径失效
- 对 `-310`、`-201`、`-301` 等关键错误码输出可定位日志
- 在白名单服务器上先执行“余额查询 -> 商品同步 -> 小额真实充值”三步验收，再扩大范围

## 14. 成功标准

本设计完成后，后续实现应满足以下标准：

1. 深圳科飞可作为正式供应商在后台完成配置。
2. 后台或内部接口可以真实查询深圳科飞余额。
3. 后台或内部接口可以真实同步深圳科飞产品目录并回写平台商品映射。
4. 现有开放下单接口可以通过深圳科飞真实完成话费充值。
5. 深圳科飞可作为目标商品的主要供应商参与现有充值主链路。
6. 部署到白名单服务器后，可通过手工调用我方 API 完成真实联调验收。
