# ISP 话费充值平台 V1 代码与文档验收比对设计

## 1. 背景

当前项目已经具备较高完成度，后端实现主要位于 `backend/`，验收口径与模块 PRD 位于 `docs/`。本次工作的目标不是直接修改业务实现，而是先建立一套稳定、可复核的“代码与文档比对验收方法”，用于判断当前软件是否已经实现 V1 有效范围内的软件功能，并识别实现缺口与文档问题。

在前置探索中，已确认当前文档同时包含以下两类信息：

- 面向目标态的 PRD 需求描述
- 面向当前交付状态的“当前实现说明”

两类信息在部分模块内存在并存甚至冲突的情况。因此，本次验收不以所有历史表述作为同等强度的验收要求，而以“当前实现说明”所代表的有效 V1 交付口径作为判定基线。

## 2. 目标

本设计用于指导一次完整的验收比对工作，并最终回答两个核心问题：

1. 当前软件功能是否已经基本覆盖文档定义的 V1 有效范围。
2. 当前文档是否足以支撑正式验收，或仍存在会误导测试、验收和交接的冲突描述。

## 3. 非目标

本次工作不包含以下内容：

- 不直接新增业务功能。
- 不对现有实现做重构或修复。
- 不把所有 README 或 PRD 的历史表述都当作强制验收项。
- 不把文档冲突自动认定为实现缺陷。

如果在比对过程中发现真实实现缺口，可在最终报告中作为缺口提出，但不在本轮直接实现。

## 4. 验收基线

### 4.1 主基线

验收基线采用“混合证据法”，其中文档侧以 `docs/*.md` 为主清单，但判定时遵循以下原则：

- 优先采用文档中的“当前实现说明”。
- 若验收标准与“当前实现说明”冲突，以“当前实现说明”为准。
- 若某条文档描述明显保留旧模型、旧接口或旧范围，且代码实现已经体现新的当前口径，则将其归类为“文档过期/冲突”，而非实现缺陷。

### 4.2 实现范围

本次验收覆盖以下实现来源：

- `backend/src/` 下的业务代码、路由、服务、契约、仓储和插件
- `backend/src/database/` 下的迁移脚本与种子数据
- `backend/tests/` 下的自动化测试
- `docs/*.md` 下的整体设计与模块 PRD
- `backend/src/modules/*/README.md` 中与当前实现相关的模块设计说明

### 4.3 模块拆分

验收按以下模块输出：

- 用户与权限
- 渠道
- 商品
- 订单
- 供应商
- 结算与账务
- 风控
- 通知
- Worker 任务系统

## 5. 比对方法

### 5.1 方法概述

本次采用“文档主清单 + 多源证据回填”的混合证据法。执行时先从 PRD 中抽取有效验收项，再回到代码、测试、数据库模型和种子数据中寻找对应证据，最后形成逐项判定。

### 5.2 证据优先级

每条验收项至少需要一类主证据，优先级如下：

1. 代码实现证据  
   包括路由、service、repository、schema、contract、migration、seed。用于证明功能客观存在。

2. 自动化测试证据  
   包括 `backend/tests/*.test.ts`。用于证明关键场景已被验证。

3. 数据结构证据  
   包括建表脚本、状态枚举、字段设计和初始化数据。用于证明功能具有稳定的持久化模型。

4. 文档交叉证据  
   包括 `docs/*.md` 与 `backend/src/modules/*/README.md`。用于判断实现与文档口径是否一致，而不是单独证明功能存在。

### 5.3 逐条判定流程

每条验收项按以下顺序处理：

1. 从模块 PRD 中提取一个有效检查项。
2. 在实现中定位对应入口，包括 HTTP 路由、服务方法、契约调用或数据库结构。
3. 检查是否存在自动化测试覆盖关键正反场景。
4. 根据证据完整度给出最终结论。

## 6. 判定规则

### 6.1 结果标签

每条验收项只允许落入以下四类结果之一：

- `已实现`
- `部分实现`
- `未实现`
- `文档过期/冲突`

其中：

- `已实现`：主流程存在，关键规则有代码依据，且最好有自动化测试或稳定落库模型支撑。
- `部分实现`：主流程存在，但关键子能力缺失、边界不完整，或验证证据明显不足。
- `未实现`：文档要求明确存在，但在代码、测试和数据库模型中都无法找到可靠实现依据。
- `文档过期/冲突`：文档内部口径冲突或仍保留旧描述，而实现符合“当前实现说明”所定义的有效范围。

### 6.2 证据强度

为了避免结论过度武断，每条验收项同时标注证据强度：

- `强`：代码 + 测试 + 数据模型
- `中`：代码 + 测试，或代码 + 数据模型
- `弱`：仅有代码痕迹，缺少足够验证或持久化支撑

## 7. 输出结构

最终验收报告采用以下固定结构：

### 7.1 总体结论

直接回答：

- 当前软件功能是否基本覆盖 V1 有效范围
- 当前文档是否足以支撑正式验收

### 7.2 模块验收矩阵

按模块输出以下字段：

- 验收项
- 判定结果
- 证据强度
- 关键证据位置
- 备注

### 7.3 关键缺口清单

只列真正影响验收的实现缺口，并按严重度分级：

- `P1`：直接影响主链路或核心验收项，足以判定模块未达标
- `P2`：不一定阻塞上线，但会导致某项验收标准无法完整闭环
- `P3`：不影响 V1 核心功能成立，但影响可验证性、可运维性或文档准确性

### 7.4 文档问题清单

单独列出以下问题：

- 文档内部自相矛盾
- PRD 仍保留旧模型或旧接口
- 验收标准超出“当前实现说明”
- README 与 PRD 口径不一致

### 7.5 建议结论

最终建议只允许以下三种结论之一：

- `可验收通过`
- `有条件通过`
- `暂不建议通过`

其使用规则如下：

- `可验收通过`：核心模块无 P1 缺口，存在的 P2/P3 不影响 V1 主目标成立
- `有条件通过`：主能力基本齐全，但存在少量 P2 缺口或较多文档问题，需要补充说明后再正式验收
- `暂不建议通过`：核心主链路存在 P1 缺口，或文档与实现偏差过大，无法形成统一验收口径

## 8. 执行顺序

正式执行验收时，按以下顺序推进：

1. 读取 `docs/*.md`，提取有效验收项。
2. 对照 `backend/src/`、`backend/src/database/` 和 `backend/tests/` 回填证据。
3. 对每个模块输出逐项判定。
4. 汇总实现缺口与文档问题。
5. 输出总体结论与建议结论。

## 9. 成功标准

本设计完成后，后续验收执行应满足以下标准：

- 任何一项“已实现”结论都能被明确证据追溯。
- 任何一项“未实现”结论都能说明缺失点落在何处。
- 文档问题与实现问题被清晰分开，不相互污染。
- 最终结论可以被研发、测试和产品共同复核，而不是依赖单点主观判断。
