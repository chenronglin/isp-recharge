# ISP 话费充值平台角色满足度审查与下一步工作计划

## 1. 文档目的

本文基于 [docs/整体设计.md](docs/整体设计.md) 中定义的 7 个角色，对 `docs/*.md` 与 `backend/src/modules/*` 的当前实现进行交叉审查，输出两类结论：

1. 从角色实际使用场景出发，当前文档里没有明确写出的需求。
2. 文档已经写明，但后端代码尚未实现或尚未完整实现的能力。

判定原则：

- 以 [docs/整体设计.md](docs/整体设计.md) 为最高口径。
- 模块 PRD 与总设计冲突时，优先记为“文档待对齐”，不直接认定为代码缺陷。
- 当前 `backend` 为单体模块化实现，内部 contract 与事件总线属于有效实现形态，不以“必须拆成 HTTP 微服务”为判定标准。

## 2. 总体结论

- 主交易链路已经基本闭环：开放下单、手机号识别、商品匹配、渠道校验、余额扣款、供应商履约、失败退款、终态通知、基础对账、Worker 调度都有落点。
- 目前主要问题不在“主链路缺失”，而在“角色闭环不完整”和“运维治理能力不足”。
- 文档侧最大问题是：部分角色需要的后台操作和排障能力没有明确写出来。
- 后端侧最大问题是：权限与审计、周期任务、供应商治理、财务补偿与后台检索能力还不完整。

## 3. 角色满足度审查

### 3.1 平台超级管理员

当前已有支撑：

- 后台登录、刷新、登出。
- 用户与角色基础数据模型。
- 部分后台写接口已写审计日志。

文档未明确写出的需求：

- 用户停用/启用。
- 给用户分配角色。
- 查询审计日志。
- 对敏感操作的审批与留痕口径。

文档有但代码未完整实现：

- 角色只体现在 JWT 中，后台路由尚未基于角色做访问控制。
- “所有后台写操作必须记录审计日志”尚未落实为全覆盖。
- 缺少用户角色分配接口、用户状态管理接口、审计日志查询接口。
- 连续登录失败锁定、登录日志留存尚未实现。

### 3.2 平台运营

当前已有支撑：

- 渠道、凭证、授权、价格、限额、回调配置管理。
- 商品列表、商品创建、商品更新。
- 供应商列表、供应商配置、供应商余额查询、手工目录同步、同步日志查询。
- 订单列表、订单详情、订单事件、异常标记、备注、重试通知。

文档未明确写出的需求：

- 供应商余额查询。
- 手工触发供应商目录同步。
- 查询供应商同步日志。
- 商品新增与编辑入口。

文档有但代码未完整实现：

- 缺少供应商对账差异查询接口。
- 缺少供应商熔断恢复接口。
- 自动全量/动态同步没有按文档节奏持续运行。
- 原始供应商请求/响应日志未完整落库。
- 供应商健康治理与评分路由未按文档完整实现。

### 3.3 平台财务

当前已有支撑：

- 账户列表。
- 流水列表。
- 渠道余额充值。
- 下单扣款、失败退款、基础价差记账。

文档未明确写出的需求：

- 渠道余额充值/调账入口。
- 财务动作的审计要求。

文档有但代码未完整实现：

- 缺少 `GET /admin/ledger-entries/:entryId`。
- “退款失败后由 Worker 自动补偿重试”尚未实现。
- 成本差异虽可通过对账差异落库表达，但财务侧的专门查询闭环仍不完整。

### 3.4 平台风控

当前已有支撑：

- 风控规则管理。
- 黑白名单管理。
- 风控决策查询。
- 下单前同步 `PASS / REJECT`。

文档未明确写出的需求：

- 角色本身在文档中已有较清晰定义，额外缺口较少。
- 更需要补的是“风控角色对后台接口的访问边界”。

文档有但代码未完整实现：

- 风控角色没有被真正用于后台访问控制。
- 风控服务异常时默认拒绝的保护策略没有明确的超时/降级实现。

### 3.5 技术支持 / 运维

当前已有支撑：

- 订单详情与事件查询。
- 通知任务与死信查询。
- Worker 任务、任务详情、重试、取消、死信查询。
- 供应商回调日志留存。

文档未明确写出的需求：

- 按订单号、手机号、供应商订单号统一检索。
- 查看供应商原始请求/响应日志。
- 查看回调原文与验签结果。
- 运维视角的差异跟踪与排障入口。

文档有但代码未完整实现：

- 后台订单列表缺少按订单号、手机号、供应商订单号过滤。
- 通知任务详情未形成“任务 + 最近投递结果”的完整排障视图。
- `supplier_request_logs` 表存在，但实际未写入。
- 供应商熔断与人工恢复能力未实现。
- Worker 周期任务当前只会启动时投递一次，不会持续重复执行。

### 3.6 商户 / 企业客户

当前已有支撑：

- 开放下单。
- 开放订单查询与事件查询。
- 开放商品列表。
- 开放签名鉴权。
- 渠道档案查询与额度查询接口已存在。

文档未明确写出的需求：

- 渠道档案查询能力。
- 渠道额度/限额查询能力。

文档有但代码未完整实现：

- 当前主要缺口不在主链路，而在文档没有把已存在的自助查询能力表达清楚。

### 3.7 上游供应商

当前已有支撑：

- 供应商回调入口。
- 供应商提单、查单、目录同步适配层。
- 深圳科飞真实供应商接入路径。

文档未明确写出的需求：

- 应明确供应商侧只暴露平台接收回调入口，不承诺供应商自助门户或额外自助 API。

文档有但代码未完整实现：

- 供应商内部同步/对账 HTTP 入口在 PRD 中有列出，但当前实现主要靠内部模块与 Worker 编排，需文档对齐。

## 4. 角色视角下文档缺口汇总

以下内容应补入相关 PRD、README 或总设计附录，避免角色职责和后台能力脱节：

1. 超级管理员：
   - 用户停用/启用
   - 用户角色分配
   - 审计日志查询
   - 敏感操作审批与留痕规则

2. 平台运营：
   - 供应商余额查询
   - 手工目录同步
   - 同步日志巡检
   - 商品新增/编辑入口

3. 平台财务：
   - 渠道余额充值/调账
   - 财务操作审计要求

4. 技术支持 / 运维：
   - 多维检索入口
   - 回调原文排障
   - 供应商请求日志排障
   - 差异跟踪入口

5. 商户 / 企业客户：
   - 渠道档案查询
   - 渠道额度/限额查询

6. 上游供应商：
   - 明确仅提供回调接入，不提供自助管理入口

## 5. 文档已有但后端未实现的缺口汇总

### 5.1 IAM / 权限与审计

- 角色级访问控制未落地。
- 审计日志未覆盖所有后台写接口。
- 缺少用户角色分配接口。
- 缺少用户停用/启用接口。
- 缺少审计日志查询接口。
- 缺少连续登录失败锁定。
- 缺少登录日志留存能力。

### 5.2 Worker / 调度系统

- 周期任务不是“真正周期执行”，只是启动时投递一次。
- 缺少退款失败补偿重试任务。

### 5.3 Suppliers / 供应商治理

- 候选评分仅按 `priority + costPrice`，未实现成功率、稳定性、利润、最近耗时排序。
- 熔断、恢复、降权未实现。
- 缺少 `GET /admin/suppliers/reconcile-diffs`。
- 缺少 `POST /admin/suppliers/:supplierId/recover-circuit-breaker`。
- 缺少文档声明的内部同步/对账 HTTP 入口，或至少未在文档中声明“当前采用模块内编排替代”。
- `supplier_request_logs` 未被实际写入。
- 自动全量/动态同步未按文档频率持续执行。

### 5.4 Ledger / 账务

- 缺少 `GET /admin/ledger-entries/:entryId`。
- 退款失败自动补偿重试未实现。

### 5.5 Products / 商品

- 动态数据超 120 分钟后只是匹配时过滤，未正式落成“库存维护”状态切换。
- 文档未及时同步当前已经存在的商品创建/编辑后台能力。

### 5.6 Orders / 运维查询

- 缺少按订单号、手机号、供应商订单号快速定位的后台过滤能力。

### 5.7 Notifications / 通知

- 任务详情未包含完整投递日志视图，排障信息不足。

### 5.8 文档待对齐项

- [docs/渠道服务PRD.md](docs/渠道服务PRD.md) 中“渠道可按层级创建并维护上下级关系”与总设计“单层渠道主体”冲突，应先修文档口径。

## 6. 下一步工作流

### 工作流 A：文档对齐

目标：

- 先统一总设计、各模块 PRD 与模块 README 的角色口径、接口清单和范围边界。

具体事项：

1. 修正渠道服务文档中的“多层级渠道”表述，统一为单层渠道。
2. 把商品创建/编辑、供应商余额查询、手工同步、同步日志、渠道档案/额度查询补入相应文档。
3. 在供应商与通知文档中说明当前哪些内部能力通过模块编排实现，而非独立 HTTP 服务。
4. 把技术支持/运维的排障需求补充为独立条目。

### 工作流 B：权限与审计补齐

目标：

- 先补“谁能做什么”和“做了什么可追溯”。

具体事项：

1. 增加后台角色访问控制中间层。
2. 新增用户角色分配接口。
3. 新增用户停用/启用接口。
4. 新增审计日志查询接口。
5. 给所有后台写接口补齐统一审计日志。
6. 增加登录失败次数、账号锁定与登录日志。

### 工作流 C：供应商治理与调度补齐

目标：

- 先补持续运行能力，再补供应商治理和排障。

具体事项：

1. 将 Worker 周期任务改为真正 recurring 执行。
2. 给供应商提单、查单写入 `supplier_request_logs`。
3. 新增供应商对账差异查询接口。
4. 新增供应商熔断恢复接口。
5. 实现熔断计数、临时剔除、人工恢复。
6. 把路由评分扩展为成功率、稳定性、成本、利润、优先级、最近耗时。
7. 让全量/动态同步支持按文档节奏持续执行，或明确受限原因。

### 工作流 D：账务与运维闭环补齐

目标：

- 让财务补偿和运维排障形成完整闭环。

具体事项：

1. 新增 `GET /admin/ledger-entries/:entryId`。
2. 增加退款失败补偿任务及 Worker 重试链路。
3. 给后台订单列表增加 `orderNo / mobile / supplierOrderNo` 过滤。
4. 给通知任务详情补充最近投递记录。
5. 视需要补充供应商回调与请求日志的后台查询入口。

## 7. 建议实施顺序

建议按以下优先级推进：

1. 文档对齐
2. 权限与审计补齐
3. Worker 周期调度修正
4. 账务补偿与后台检索
5. 供应商请求日志、对账差异、熔断恢复
6. 供应商评分路由优化
7. 自动同步节奏补齐

原因：

- 文档口径不先统一，后续开发会继续反复返工。
- 权限与审计属于平台底座，越晚补成本越高。
- 周期任务不是真正周期执行，会直接影响超时扫描、对账、运维发现能力。

## 8. 验收建议

### 8.1 角色验收

- `SUPER_ADMIN / OPS / FINANCE / RISK / SUPPORT` 分别验证可访问与不可访问接口矩阵。
- 商户侧验证开放下单、订单查询、商品查询、渠道档案/额度查询。

### 8.2 审计验收

- 所有后台写接口都写入 `iam.operation_audit_logs`。
- 审计日志可按用户、动作、资源、时间查询。

### 8.3 Worker 验收

- `order.timeout.scan`、`supplier.reconcile.inflight`、`supplier.reconcile.daily` 在首次执行后仍会继续按周期运行。

### 8.4 供应商验收

- 请求日志、回调日志、对账差异都能落库并查询。
- 熔断命中后路由剔除，人工恢复后重新参与路由。
- 候选路由按成功率、稳定性、成本、利润、优先级生效。

### 8.5 财务验收

- 退款失败进入补偿任务，重试成功后订单完成 `REFUNDED`。
- `GET /admin/ledger-entries/:entryId` 能返回单笔流水详情。

### 8.6 运维验收

- 后台能按订单号、手机号、供应商订单号快速定位订单。
- 通知详情可查看最近投递结果。

## 9. 备注

- `shenzhen-kefei` 已可视作“至少 1 家真实供应商接入”的有效证据，后续重点应转向治理能力、角色闭环与运维能力。
- 当前重点不是推翻既有主链路，而是把已经具备主链路能力的系统补齐到“可持续交付、可运维、可审计”的状态。
